Static Page

특별한 데이터 변화나 요청 기반의 정보가 필요하지 않은 페이지는 기본적으로 정적 페이지로 처리된다. 즉, 동적 요소가 없으면 Next.js는 자동으로 해당 페이지를 정적으로 간주하고 빌드 시점에 결과물을 캐싱한다. 대부분의 정적 페이지는 Full Route Cache의 혜택을 받는다.

Dynamic Page

반대로, 캐싱이 불가능한 형태의 데이터 패칭을 사용하거나 요청마다 달라지는 값—예를 들면 쿠키, 헤더, 쿼리 파라미터 등—을 사용하는 컴포넌트가 포함된 경우 페이지는 동적 페이지로 처리된다.
그래도 캐시가 완전히 사라지는 것은 아니고, 페이지 전체를 캐싱하는 대신 요청 순간의 데이터를 기반으로 새로 렌더링한다.

Full Route Cache 동작 방식

Next.js는 빌드 과정에서 정적으로 처리할 수 있는 페이지를 먼저 생성해두고, 그 결과물을 Full Route Cache에 저장한다.
사용자가 페이지에 접속하면 이 캐시된 HTML이 즉시 반환되어 서버 부하를 줄이고 렌더링 속도를 크게 높인다.
또한 revalidate 옵션을 설정할 경우, 기존의 ISR처럼 특정 시간 간격으로 데이터 및 페이지 캐시가 갱신되도록 할 수 있다.

generateStaticParams

URL 파라미터가 필요한 라우트에서 미리 어떤 값을 사용해 빌드할지 지정하는 함수다. 이 함수가 존재하면, 내부에서 동적 데이터 패칭을 사용하더라도 해당 페이지는 강제로 정적 페이지로 변환된다. Page Router의 getStaticPaths와 거의 동일한 역할을 한다.
특이점은 파라미터 값이 문자열 형태로 제공되어야 한다는 점이다.

Route Segment Options

특정 페이지의 동작 방식을 개발자가 직접 제어할 수 있게 하는 설정이다. 예를 들어 “이 페이지는 강제로 동적 처리해야 한다”거나 “캐싱을 사용하면 안 된다” 같은 동작을 지정할 수 있다. dynamicParams 옵션도 그중 하나다.

Client Router Cache

브라우저 단에서 관리되는 캐시로, 페이지 전환을 더 빠르게 만들기 위해 일정 데이터를 로컬에 저장한다. 서버 측 Full Route Cache와 달리 사용자 환경에서 동작하며, React의 클라이언트 라우팅을 효율화하는 데 사용된다.
